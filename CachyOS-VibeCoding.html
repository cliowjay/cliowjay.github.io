觀念篇

先搞懂：這四個工具各自負責什麼

名字很像、很容易混淆。花兩分鐘看懂它們的關係，後面安裝就不會迷路。

工具角色比喻它負責的事
Ollama發電廠在你的電腦上「執行」AI 模型的軟體。負責把模型載入顯示卡、提供一個本機網址讓其他程式呼叫。
Qwen 3.5電力本身真正的 AI 模型（阿里巴巴出品）。由 Ollama 來執行它。有大有小，挑適合 8GB 顯示卡的尺寸。
Cline桌上的助理VS Code 的擴充套件。把 Qwen 3.5 接進編輯器，讓 AI 直接幫你讀檔、寫檔、執行指令 —— 這就是「Vibe Coding」。
OpenClaw一位全能員工一個會「主動行動」的 AI 代理：會自己排程、有長期記憶、能操作你的檔案與工具。一個人、一個代理。
HiClaw一整間公司OpenClaw 的「團隊版」。一個 Manager 代理指揮多個 Worker 代理協作，所有對話都在聊天室裡，你隨時能介入。
ℹ️ 它們是怎麼串起來的
Ollama 跑 Qwen 3.5 → 提供一個本機 AI 服務 → Cline / OpenClaw / HiClaw 都來呼叫這個服務當大腦。
換句話說：階段一裝好的這顆「本地大腦」，階段二、三、四全部共用它。而 HiClaw 內部其實就是用 OpenClaw 當每個代理的執行核心 —— 所以階段三先單獨玩 OpenClaw，是為了理解原理；階段四的 HiClaw 則把它變成一支隊伍。
⚠️ 關於模型大小與你的 8GB 顯示卡
顯示卡記憶體（VRAM）只有 8GB，模型必須塞得進去才會快。本指南主力推薦 Qwen 3.5 9B（約 6GB，整顆進得了 GPU）。它對「寫程式聊天」很夠用，但對「完全自主、多步驟的代理任務」算是偏小的模型 —— 這點我們會在每個階段誠實提醒你，並給出對策。
前置作業

開始之前：系統準備

確認驅動程式就緒、系統更新到最新。大約 5～10 分鐘。

CachyOS 是「滾動更新」的系統，安裝新軟體前先把所有套件更新到最新，可以避免很多相容性問題。

sudo pacman -Syu
這行指令在做什麼？
pacman 是 CachyOS（Arch 系）的「軟體管理員」，負責安裝/移除/更新軟體。-S 是同步、-y 是刷新軟體清單、-u 是升級已安裝的套件。看到問你 [Y/n] 時按 Enter 即可。若清單很長、跑幾分鐘都正常。

Qwen 3.5 要靠顯示卡加速，所以驅動一定要先就緒。執行這行：

nvidia-smi

如果出現一張表格，裡面看得到 NVIDIA RTX A2000 字樣 —— 太好了，驅動正常，跳過下一個步驟。

如果出現「command not found」或錯誤訊息，代表驅動還沒裝好，繼續看步驟 0-3。

nvidia-smi 是什麼？
它是 NVIDIA 內附的監控工具，會顯示顯示卡型號、溫度、VRAM 使用量、正在用 GPU 的程式。之後想知道「模型有沒有真的在用顯示卡」也是靠它。

CachyOS 安裝時通常會自動裝好顯示卡驅動，所以多數情況你不需要這步。若上一步失敗才執行：

sudo pacman -S nvidia-open-dkms nvidia-utils

裝完後重新開機，再執行一次 nvidia-smi 確認。

ℹ️ 為什麼用 nvidia-open
RTX A2000 是 Ampere 架構，NVIDIA 的「開源核心模組」驅動（nvidia-open）完整支援，且和 CachyOS 的滾動更新搭配較穩定。

替代做法：CachyOS 內建硬體偵測工具，也可執行 sudo chwd -a 讓系統自動挑驅動安裝。

後面會用到 git 與編譯工具，先一次裝齊。--needed 代表已經有的就略過。

sudo pacman -S --needed base-devel git curl
✅ 完成這一段後，你應該能：
  • 執行 nvidia-smi 看到 RTX A2000
  • 系統已更新到最新
  • git 已安裝（git --version 有回應）
階段一

Ollama + Qwen 3.5：讓本地 AI 跑起來

1
這一階段的目標
裝好 Ollama，下載 Qwen 3.5 模型，確認它跑在顯示卡上，並調好效能設定。約 20～30 分鐘（多數時間在下載模型）。
🚨 最關鍵的一步，別裝錯套件
一定要裝 ollama-cuda，不是普通的 ollama。普通版只會用 CPU 跑，慢上好幾倍。ollama-cuda 才會用到你的 RTX A2000。
sudo pacman -S ollama-cuda
CUDA 是什麼？
CUDA 是 NVIDIA 顯示卡的運算平台。AI 模型的計算量極大，靠顯示卡上千個核心平行處理會比 CPU 快非常多。ollama-cuda 就是「會使用 CUDA 的 Ollama 版本」。

讓 Ollama 變成開機自動啟動的背景服務，這樣它會一直待命等你呼叫。

sudo systemctl enable --now ollama

確認版本，有版本號就代表裝好了：

ollama --version
systemctl enable --now 是什麼意思？
systemctl 管理系統的背景服務。enable 設定為「開機自動啟動」，--now 代表「順便現在就立刻啟動」。所以這一行同時做了兩件事。

下載主力模型 Qwen 3.5 9B（約 6GB，會邊下載邊顯示進度）：

ollama pull qwen3.5:9b

再下載一顆輕量模型 Qwen 3.5 4B，需要快速回應時用：

ollama pull qwen3.5:4b
模型下載大小放哪適合做什麼
qwen3.5:9b約 6 GB整顆進 GPU ✅主力。Vibe coding、代理日常任務
qwen3.5:4b約 2.5 GB整顆進 GPU ✅快速問答、簡單任務
qwen3.5:27b約 15 GB部分溢出到 RAM ⚠️較複雜的推理，但會變慢
qwen3.5:35b-a3b約 20 GB部分溢出到 RAM ⚠️MoE 架構，溢出也還能接受的較強模型
💡 建議
先用 9B 走完整份指南。等熟悉之後想要更強的推理品質，再試 qwen3.5:35b-a3b —— 它是 MoE（混合專家）架構，雖然 8GB 裝不下、要溢出一部分到 RAM，但每次只啟用一小部分參數，所以速度比同級的 27B 一般模型好。
「pull」是什麼？下載到哪裡？
pull（拉取）會從 Ollama 官方模型庫把模型檔下載到你電腦（預設在 /usr/share/ollama）。下載一次就好，之後都從本機讀取。若中途斷線，重跑同一行指令會「續傳」，不會從頭來。

啟動 9B 模型，進入對話模式：

ollama run qwen3.5:9b

看到 >>> 提示符後，隨便打個問題試試，例如「用 Python 寫一個費氏數列函式」。要離開對話，輸入 /bye 再按 Enter。

這步很重要 —— 要確認模型用的是 GPU 而不是 CPU。模型載入狀態下，開「另一個終端機視窗」執行：

ollama ps

看 PROCESSOR 那一欄。顯示 100% GPU 就完美。若顯示 CPU 或部分 CPU，請看下方「疑難排解」。

也可以用 nvidia-smi 看：模型載入時，VRAM 用量應該會跳到約 6GB 左右。

✅ 階段一驗收
  • ollama --version 有版本號
  • 能和 qwen3.5:9b 對話
  • ollama ps 顯示 100% GPU

幫 Ollama 加上幾個環境設定，讓 8GB VRAM 用得更有效率，也順便為後面的 HiClaw 預作準備。執行：

sudo systemctl edit ollama

會打開一個文字編輯畫面。在指定的空白區域（通常標示著「### 在此行與結尾之間加入你的設定」）貼上以下內容：

[Service]
Environment="OLLAMA_FLASH_ATTENTION=1"
Environment="OLLAMA_KV_CACHE_TYPE=q8_0"
Environment="OLLAMA_NUM_PARALLEL=2"
Environment="OLLAMA_MAX_LOADED_MODELS=1"
Environment="OLLAMA_KEEP_ALIVE=10m"
Environment="OLLAMA_CONTEXT_LENGTH=32768"
Environment="OLLAMA_HOST=0.0.0.0:11434"

存檔離開（編輯器若是 nano：Ctrl+O 存檔、Enter 確認、Ctrl+X 離開）。然後重啟服務讓設定生效：

sudo systemctl restart ollama
設定作用
FLASH_ATTENTION=1開啟更省記憶體的注意力演算法，讓 VRAM 撐得住更長的上下文。
KV_CACHE_TYPE=q8_0把「對話快取」壓縮成 8-bit，省下大量 VRAM，品質幾乎無損。
NUM_PARALLEL=2同時最多處理 2 個請求。多代理時有用，但別設太大以免吃爆 VRAM。
MAX_LOADED_MODELS=1同時只保留 1 個模型在 VRAM。8GB 放不下兩顆，這樣最穩。
KEEP_ALIVE=10m模型用完後在記憶體裡待命 10 分鐘，免得每次都重新載入。
CONTEXT_LENGTH=32768預設上下文長度 32K，對寫程式夠用。設太大會吃爆 VRAM。
HOST=0.0.0.0:11434讓 Ollama 接受來自其他程式（含階段四的 Docker 容器）的連線。HiClaw 會用到。
⚠️ 安全小提醒
OLLAMA_HOST=0.0.0.0 會讓同一個區域網路內的裝置都能連到你的 Ollama。在家用網路沒問題；如果在公司或公共網路、且有疑慮，可改成只開放給 Docker：OLLAMA_HOST=172.17.0.1:11434。
階段二

Cline：在 VS Code 裡開始 Vibe Coding

2
這一階段的目標
在你天天用的 VS Code 裡裝上 Cline，接上本地 Qwen 3.5，第一次體驗 AI 幫你寫整個專案。約 15 分鐘。
ℹ️ 為什麼推薦 Cline（這是你問我要的推薦）
你做 vibe coding 的選擇主要有三個：Cline、Continue.dev、Aider。我推薦 Cline，因為：
① 它是「agentic（代理式）」工具 —— 會自己讀檔、改檔、執行終端機指令、看結果再修正，最貼近「用嘴巴寫程式」的 vibe coding 體驗。
② 它是 VS Code 擴充，完全在你已經熟悉的環境裡，零學習成本。
③ 它的運作模式（規劃 → 執行 → 驗證）正好是階段三、四 AI 代理的縮影，先用 Cline 培養手感，後面接觸 OpenClaw/HiClaw 會非常順。
Continue.dev 比較像「程式碼補全＋側邊對話」，較輕量；Aider 是純終端機工具。若之後覺得 9B 模型帶 Cline 有點吃力，Continue.dev 是很好的備援。

打開 VS Code，按 Ctrl+Shift+X 開啟擴充套件面板，在搜尋框輸入 Cline，找到後點「Install / 安裝」。

安裝完，VS Code 左側活動列會出現一個 Cline 圖示（機器人圖案），點它就會開啟 Cline 面板。

找不到 Cline？
如果你的 VS Code 是 CachyOS 套件庫安裝的開源版「Code - OSS」，它用的是 Open VSX 套件市集 —— Cline 在上面也有，照樣搜得到。若真的搜不到，可改裝微軟官方版 VS Code（AUR 套件 visual-studio-code-bin）。

點開 Cline 面板，進入設定（齒輪圖示），照下表填寫：

設定項目填入的值
API Provider選擇 Ollama
Base URLhttp://localhost:11434
Modelqwen3.5:9b

儲存後，Cline 應該就能列出你下載過的模型。挑 qwen3.5:9b 即可。

💡 小技巧
Cline 設定裡如果有「Context Window / 上下文長度」欄位，填 32768，和我們在階段一設的一致。

建一個空資料夾，用 VS Code 開啟它（檔案 → 開啟資料夾）。打開 Cline 面板，在輸入框試著下這個任務：

# 把這段話貼到 Cline 對話框
請幫我做一個單一 HTML 檔的待辦事項小工具，
用 HTML + CSS + JavaScript 寫在同一個檔案，
可以新增、勾選完成、刪除項目，介面要簡潔好看。

送出後觀察 Cline 怎麼運作：它會先「規劃（Plan）」要做什麼，列出步驟，然後一步步「執行（Act）」—— 建立檔案、寫入程式碼。每個動作它都會問你核准，你按核准它才會動手。完成後直接在瀏覽器打開那個 HTML 檔看成果。

Plan 模式 vs Act 模式
Cline 有兩種模式。Plan（規劃）：只討論、列計畫，不動你的檔案 —— 適合先把需求講清楚。Act（執行）：實際讀寫檔案、跑指令。建議養成習慣：先在 Plan 模式把需求聊到位，再切到 Act 讓它動手。對 9B 這種中小型模型，這個習慣特別能提升成功率。

9B 是顆好用但「中小型」的模型。誠實說，它不會像雲端大模型那樣每次都完美。掌握這些技巧，體驗會好很多：

  • 任務切小：與其「做一個完整的電商網站」，不如拆成「先做商品列表頁」「再做購物車」。
  • 講清楚、給脈絡：明確說用什麼語言、框架、檔名。模型猜得越少，做得越準。
  • 善用 Plan 模式：先讓它規劃，你檢查計畫沒問題再執行。
  • 一次一個檔案的修改：大範圍跨檔重構對小模型較吃力。
  • 它卡住時：把錯誤訊息貼回去給它，或換 qwen3.5:35b-a3b 試試。
💡 進階選項
如果你發現某些複雜任務本地模型實在吃力，Cline 也支援雲端 API。可以「日常用本地 9B、難題切雲端」混搭 —— 兼顧隱私、成本與能力。
✅ 階段二驗收
  • Cline 已安裝並接上 qwen3.5:9b
  • 成功讓 Cline 生成一個可運作的小程式
  • 理解 Plan / Act 兩種模式的差別
階段三

OpenClaw：你的第一個 AI 代理

3
這一階段的目標
裝一個會主動行動、有記憶的個人 AI 代理，理解「代理（Agent）」和「聊天機器人」的差別。這是通往 HiClaw 團隊的橋樑。約 15～20 分鐘。
ℹ️ Cline 和 OpenClaw 差在哪？
Cline 活在 VS Code 裡、由你一句句驅動，專注寫程式。OpenClaw 則是獨立執行的代理：它有長期記憶、能排程（例如每天早上自動做某事）、能連接你的檔案、行事曆、瀏覽器等。把 Cline 想成「結對程式設計夥伴」，OpenClaw 想成「全職員工」。

OpenClaw 用 Node.js 撰寫，需要 Node 22.16 以上（建議 24）。Arch 系套件庫的版本夠新：

sudo pacman -S nodejs npm

確認版本（應顯示 v22 或更高）：

node --version
Node.js 和 npm 是什麼？
Node.js 讓 JavaScript 程式能在電腦上（而非瀏覽器裡）執行。npm 是它的套件管理員，用來下載安裝像 OpenClaw 這類用 JavaScript 寫的工具。

用 npm 全域安裝 OpenClaw：

npm install -g openclaw@latest
⚠️ 如果出現權限錯誤（EACCES）
代表 npm 全域目錄需要權限。最簡單的替代方案是改用 OpenClaw 官方安裝腳本，它會自動處理 Node 與路徑設定： 
curl -fsSL https://openclaw.ai/install.sh | bash
「-g」是什麼意思？這樣安全嗎？
-g（global）代表裝成「全系統都能用的指令」，這樣你在任何資料夾都能輸入 openclaw。
至於上面那行 curl ... | bash：它是從 OpenClaw 官方網域下載安裝腳本來執行，這是該專案官方文件正式建議的安裝方式。執行任何 curl | bash 前都該確認來源是官方網域 —— 這裡的 openclaw.ai 即是。

OpenClaw 有一個互動式引導程式，會帶你設定。執行：

openclaw onboard --install-daemon

過程中當它問到「模型供應商（Provider）」時，這樣選：

問題選擇 / 填入
選擇 Provider從清單中選 Ollama
選擇模式選 Local only（純本地）
Ollama base URLhttp://localhost:11434
選擇模型從偵測到的清單選 qwen3.5:9b
🚨 重要：網址結尾「不要」加 /v1
OpenClaw 連本地 Ollama 時，網址要用 http://localhost:11434，千萬不要寫成 .../v1。加了 /v1 會走 OpenAI 相容模式，導致「工具呼叫」失效 —— 代理就不會動手做事了。這是官方文件特別強調的雷點。
--install-daemon 是什麼？
daemon（守護程序）是常駐背景的服務。這個參數會把 OpenClaw 的 Gateway 設成 systemd 背景服務，讓它持續執行、能排程任務 —— 這正是「代理」會主動行動的關鍵。

明確指定 OpenClaw 用 Qwen 3.5 9B 當大腦：

openclaw models set ollama/qwen3.5:9b

跑幾個檢查指令確認一切就緒：

openclaw --version
openclaw doctor
openclaw gateway status
openclaw models list --provider ollama

openclaw doctor 會體檢設定有沒有問題；gateway status 確認背景服務在跑；models list 應該列得出你的 Qwen 模型。

現在和你的代理對話、給它一個小任務，感受它「自己動手」的過程。例如請它在某個資料夾裡整理檔案、或寫一個小腳本。觀察它如何規劃、執行、回報。

💡 心態建議
這個階段重點不是「做出多厲害的東西」，而是建立對「代理」的直覺：它會自己拆解任務、呼叫工具、看結果再決定下一步。理解了這個迴圈，階段四的 HiClaw 團隊你就會秒懂 —— 因為那只是「很多個這樣的代理一起合作」。

階段四 · 最終目標

HiClaw：打造軟體開發 AI 團隊

4
這一階段的目標
用 HiClaw 架起一個 Manager 帶 Worker 的多代理團隊，建立前端、後端、測試等角色，指派一個真實的開發任務讓它們協作。約 30～40 分鐘。

先理解 HiClaw 的運作方式

HiClaw 是「協作式多代理作業系統」。核心概念：

  • Manager（管理者）：你的 AI 幕僚長。你跟它說需求，它負責建立 Worker、拆解任務、分派、追蹤進度、回報。
  • Worker（工作者）：實際幹活的代理，例如「前端工程師 alice」「後端工程師 bob」。
  • 聊天室介面（Matrix / Element）：所有對話都在聊天室裡進行，你全程看得到、隨時能插話介入 —— 沒有黑箱。
  • Higress 閘道：集中保管 API 金鑰、GitHub Token。Worker 只拿到臨時憑證，碰不到真正的金鑰，比較安全。
ℹ️ HiClaw 和階段三的 OpenClaw 是什麼關係？
HiClaw 內部其實是用 OpenClaw（或更輕量的 CoPaw）當每個代理的執行核心，並用 Docker 容器把它們包起來、加上聊天室、檔案共享與安全閘道。所以階段三玩過的 OpenClaw 不是白費 —— 它正是 HiClaw 裡每個 Worker 的「本體」。

HiClaw 用 Docker 容器來執行各個元件。安裝並啟動 Docker：

sudo pacman -S docker docker-compose
sudo systemctl enable --now docker

把你的使用者加入 docker 群組，這樣不用每次都打 sudo：

sudo usermod -aG docker $USER
⚠️ 這一步做完一定要重新登入
群組變更要重新登入（或重開機）才生效。快速生效可執行 newgrp docker，或乾脆登出再登入一次。

驗證 Docker 正常：

docker run --rm hello-world
Docker 是什麼？
Docker 把軟體連同它需要的環境打包成「容器」，像獨立的小盒子。HiClaw 有 Higress、Matrix 伺服器、檔案儲存、各個代理等很多元件，用容器隔開可以乾淨地一起執行、不互相干擾。

安裝 HiClaw 前先做個重要決定：團隊的代理要用「本地 Qwen 3.5」還是「雲端 Qwen API」？兩條路都可以，誠實的比較如下：

路線 A：雲端 Qwen API（推薦）路線 B：本地 Ollama（全免費）
設定難度簡單，貼上金鑰即可較繁瑣（要處理 Docker 網路）
推理品質較強，適合 Manager 拆解複雜任務受限於 9B 模型
多代理速度各代理並行不卡共用一張顯示卡，請求會排隊
費用 / 隱私依用量計費；資料送雲端完全免費；資料不出門
💡 我的建議
多代理協作對「大腦」要求較高，而你只有單張 8GB 顯示卡要同時餵好幾個代理會排隊。建議第一次先用路線 A（雲端 Qwen API） 把 HiClaw 跑順、把流程學會；之後若想挑戰全本地，再回來試路線 B。
折衷做法：Manager 用雲端（負責動腦拆解），Worker 用本地（負責執行）—— 兼顧品質與成本。

執行官方安裝腳本，它會用互動問答帶你完成設定：

bash <(curl -sSL https://higress.ai/hiclaw/install.sh)

安裝過程會問你幾件事：

  • LLM 供應商與 API 金鑰：見下方依路線填寫。
  • 管理員帳號與密碼：自己設定，這是你登入聊天室的帳密。
  • 網路模式：選「local-only（僅本機）」即可。
  • 網域名稱：直接按 Enter 用預設值。
  • GitHub PAT（選填）：若要讓代理直接操作 GitHub，可填入；不確定就先跳過，之後能再加。

路線 A：填雲端 Qwen API

到阿里雲百煉（DashScope）等服務開通 Qwen 模型、取得 API 金鑰，安裝時填：

欄位填入
API Base URLhttps://dashscope.aliyuncs.com/compatible-mode/v1
API Key你的 DashScope 金鑰（sk-...）
Model你開通的 Qwen 模型名稱

路線 B：指向本地 Ollama

因為 HiClaw 跑在 Docker 容器裡，容器內的 localhost 指的是容器自己、不是你的電腦。要讓容器連到主機上的 Ollama，得用 Docker 橋接位址。先確認橋接 IP（通常是 172.17.0.1）：

ip -4 addr show docker0

然後安裝時填：

欄位填入
API Base URLhttp://172.17.0.1:11434/v1
API Key隨便填，例如 ollama（本地不驗證金鑰）
Modelqwen3.5:9b
⚠️ 路線 B 的前提
必須已完成階段一步驟 1-6 的 OLLAMA_HOST=0.0.0.0:11434 設定，Ollama 才會接受來自 Docker 容器的連線。HiClaw 連不到模型，九成是這個原因。

安裝完成後，打開瀏覽器前往：

http://127.0.0.1:18088

這是 Element Web（聊天室介面）。用你剛才設定的管理員帳密登入。登入後，Manager 代理會主動跟你打招呼、說明怎麼建立第一個 Worker。

網址用途
http://127.0.0.1:18088Element Web — 你和代理團隊對話的主要地方
http://localhost:18001Higress 主控台 — 看金鑰與流量管理
http://localhost:9001MinIO 主控台 — 看代理之間共享的檔案

確認 Manager 容器在跑：

docker ps | grep hiclaw-manager

在 Element Web 裡，跟 manager 開一個私訊對話，傳送：

請建立一個名為 alice 的 Worker，負責前端開發工作，
並給她 GitHub MCP 的存取權限。

Manager 會自動：幫 alice 註冊聊天室帳號 → 在 Higress 建立她的憑證 → 建立一個三方聊天室（你、Manager、Alice）→ 啟動 Alice 的容器。完成後你會在左側看到 Alice 的房間。

什麼是 MCP？
MCP（Model Context Protocol）是讓 AI 代理連接外部工具的標準介面。「GitHub MCP」就是讓代理能讀寫 GitHub 儲存庫、開分支、發 PR 的工具。金鑰由 Higress 閘道集中保管，Worker 只透過 MCP 呼叫、碰不到真金鑰。

用同樣的方式請 Manager 再建立幾個角色，組成一支麻雀雖小、五臟俱全的開發隊：

Worker角色建立指令（傳給 Manager）
alice前端開發已於上一步建立
bob後端開發「建立名為 bob 的 Worker，負責後端開發，給他 GitHub MCP 存取權。」
carol測試 / QA「建立名為 carol 的 Worker，負責撰寫測試與品質檢查。」
💡 省記憶體訣竅：用 CoPaw 執行 Worker
HiClaw 的 Worker 可以用較輕量的「CoPaw」執行核心，記憶體用量約 150MB，遠低於標準 OpenClaw 的約 500MB。建立 Worker 時可以加一句「請用 CoPaw runtime 建立」。對你 32GB 的機器，這能讓你多養好幾個代理。
⚠️ 循序漸進
先從 2～3 個 Worker 開始，觀察記憶體與速度再決定要不要加人。單張顯示卡＋本地模型時，代理越多、請求排隊越明顯。

現在來真的。在和 Manager 的對話裡，下一個需要多人協作的任務：

我要 alice、bob、carol 一起開發一個簡單的記帳網頁應用：

- alice：用 React 做前端介面（新增支出、列表、月度總計）
- bob：用 Node.js + Express 做後端 API 與資料儲存
- carol：為後端 API 撰寫測試

請大家透過 MinIO 共享檔案協調介面規格（API 格式），
有重要決策時在房間裡回報，完成後通知我。

接著就觀察魔法發生：Manager 會拆解任務、分派給三位 Worker，它們在各自房間裡工作、把進度貼出來。你全程看得到，隨時能在任一房間補充指示 —— 例如「alice，按鈕改用圓角」「bob，密碼至少 8 碼」。

💡 結論

什麼時候該考慮雲端 API

全本地很棒，但在這些情況混搭雲端會明顯更順：Manager 要拆解複雜大型任務時、多個 Worker 需要同時高速運作時、或本地 9B 在某類任務上品質不夠時。建議的心法是「日常本地、難題雲端」—— 隱私、成本、能力三者取得平衡。

疑難排解

點開對應的問題看解法。

nvidia-smi 找不到指令，或沒有輸出
驅動沒裝好。回到「系統準備」步驟 0-3 安裝 nvidia-open-dkms nvidia-utils，然後務必重新開機。重開後再試一次。
ollama ps 顯示 CPU 而不是 100% GPU
最常見原因是裝成了普通 ollama 而非 ollama-cuda。先移除再裝對的： 
sudo pacman -R ollama
sudo pacman -S ollama-cuda
sudo systemctl restart ollama
若仍不行，確認 nvidia-smi 正常，並重啟 Ollama 服務。
ollama pull 下載很慢或中途斷掉
這是網路問題。好消息是 Ollama 支援「續傳」—— 直接重跑同一行 ollama pull 指令，它會接著沒下載完的部分繼續，不會從頭來。
Cline 回應很慢、或結果品質不理想
9B 是中小型模型，這在預期之內。對策：把任務切得更小更具體、先用 Plan 模式把需求講清楚、把錯誤訊息貼回去讓它修正。需要更強時改用 qwen3.5:35b-a3b，或在 Cline 裡切換到雲端 API。
Cline 連不到 Ollama / 找不到模型
確認 Ollama 服務在跑（systemctl status ollama），且 Cline 的 Base URL 是 http://localhost:11434。可用這行測試 API 是否通： 
curl http://localhost:11434/api/tags
有回傳一串 JSON 就代表 Ollama 正常。
輸入 openclaw 出現 command not found
npm 全域安裝的路徑沒被加進系統 PATH。先看路徑： 
npm prefix -g
把它的 /bin 子目錄加進 PATH。執行這行（會寫進你的 shell 設定檔）： 
echo 'export PATH="$(npm prefix -g)/bin:$PATH"' >> ~/.bashrc
然後關掉重開終端機。（若你用 zsh，把 .bashrc 改成 .zshrc。）
docker 指令出現 permission denied
你還沒被加進 docker 群組，或加了但沒重新登入。執行 sudo usermod -aG docker $USER 後，登出再登入（或重開機）。臨時生效可用 newgrp docker。
HiClaw 連不到本地 Ollama 模型
這幾乎都是 Docker 網路問題。檢查三件事：① 階段一步驟 1-6 是否已設 OLLAMA_HOST=0.0.0.0:11434 並重啟 Ollama；② HiClaw 的 Base URL 是否用了 http://172.17.0.1:11434/v1（容器內不能用 localhost）；③ 用 ip -4 addr show docker0 確認橋接 IP 確實是 172.17.0.1。若仍卡關，最省事的方式是改用雲端 Qwen API（路線 A）。
顯示記憶體不足（out of memory / VRAM）
8GB VRAM 被塞爆了。對策：降低 OLLAMA_CONTEXT_LENGTH（例如改成 16384）、確認 OLLAMA_KV_CACHE_TYPE=q8_0 有開、確認 OLLAMA_MAX_LOADED_MODELS=1、或改用更小的 qwen3.5:4b。
想看 HiClaw 的詳細記錄檔（log）
查 Manager 代理的日誌： 
docker exec -it hiclaw-manager cat /var/log/hiclaw/manager-agent.log
看單一容器即時日誌可用 docker logs -f <容器名稱>。
網頁打不開（連接埠 18088 等）
先確認容器都起來了：docker ps。若連接埠被其他程式占用，安裝腳本通常會提示；可關掉占用的程式後重跑安裝腳本。剛安裝完也可能需要等一兩分鐘讓所有服務啟動完成。
